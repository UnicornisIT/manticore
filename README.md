# manticore

`manticore` - веб-приложение для подготовки логинов Moodle, ведения абитуриентов, загрузки студентов и контроля данных приемной кампании.

Проект сделан как локальная рабочая система для приемной комиссии или учебного отдела: его можно запустить на одном компьютере, открыть в браузере и дать доступ другим сотрудникам в локальной сети. Данные хранятся в локальной SQLite-базе, а импорт и выгрузка выполняются через Excel/CSV-файлы.

## Что умеет приложение

- Загружает абитуриентов из `.xlsx`, `.xls` и `.csv`.
- Перед импортом показывает полный предпросмотр строк и не меняет базу до подтверждения.
- Автоматически формирует логины Moodle по номеру договора.
- Позволяет настроить собственные правила генерации логинов через интерфейс администратора.
- Разделяет записи по приемным кампаниям, например `2025`, `2026`, `2027`.
- Находит дубли, повторяющиеся договоры, ошибки договоров и конфликты логинов.
- Ведет список абитуриентов с фильтрами, сортировкой, поиском, почтой, оплатой и комментариями.
- Позволяет массово отметить оплату, снять оплату, выгрузить или удалить выбранных абитуриентов.
- Обновляет почту и оплату абитуриентов отдельным файлом по договору.
- Загружает готовых студентов Moodle с предпросмотром, проверкой обязательных полей и дублей.
- Переносит готовых абитуриентов в студенты с сохранением исходного договора и кампании.
- Ведет справочник академических групп по годам и контролирует заполненность групп.
- Поддерживает авторазбиение по подгруппам, если выбранная группа переполнена.
- Показывает панель состояния, центр задач и полную проверку качества данных.
- Дает глобальный поиск по абитуриентам, студентам, дублям и конфликтам.
- Открывает карточку записи с понятными названиями полей.
- Ведет журнал действий и автоматически создает резервные копии перед важными изменениями.
- Поддерживает роли пользователей и одобрение заявок на регистрацию.
- Имеет базовые меры безопасности: CSRF-защита, хеширование паролей, ограничение попыток входа, безопасная работа с загрузками и security headers.

## Что добавили и улучшили в последних версиях

- Первичная настройка правил логинов: администратор после первого входа должен выбрать стандартный вариант или сохранить свои правила.
- Конструктор правил логинов: шаблон, длина номера, префиксы ошибок и дублей, область уникальности, регулярное выражение года, коды специальностей и базы.
- Предпросмотр импорта абитуриентов: таблица показывает все строки файла, будущие действия, статусы и возможных тезок.
- Отчеты по строкам файла: ошибки показываются с номером строки и понятным объяснением.
- Предпросмотр загрузки студентов: база студентов меняется только после подтверждения.
- Обновление почты и оплаты отдельным файлом: данные обновляются по столбцу `Договор`.
- Панель состояния: вместо разрозненных счетчиков есть центр задач и сводка по текущей кампании.
- Полная проверка данных: отдельная страница группирует проблемы по абитуриентам, студентам, дублям и конфликтам.
- Глобальный поиск в верхнем меню и отдельная карточка записи.
- Справочник групп по папкам года, скрытие групп, CSV-импорт, шаблон CSV и автоматическое создание следующих подгрупп.
- Мастер миграции абитуриентов в студенты с проверкой готовности, групп и конфликтов.
- Архивирование кампаний: архивная кампания остается доступной для просмотра, но закрывается для изменений.
- Резервные копии базы и восстановление из бэкапа через админку.
- Журнал действий администратора и пользователей.
- Улучшенная безопасность входа: пароли хранятся в виде хеша, старые пароли мигрируют автоматически, неудачные попытки входа ограничиваются.
- Обновлены зависимости: проект использует Flask 3.x, `pandas`, `openpyxl`, `xlrd`, `python-dotenv`, а в продакшене может запускаться через `gunicorn`.
- Добавлены файлы для Linux-развертывания через `systemd` и Nginx, пример HTTPS-конфига и workflow `pip-audit`.

## Основной рабочий процесс

Типовой путь работы с новой приемной кампанией:

1. Администратор запускает приложение и входит под учетной записью `admin`.
2. Администратор настраивает правила генерации логинов.
3. Администратор выбирает активную приемную кампанию, например `2026`.
4. Сотрудник загружает файл абитуриентов на странице `Работа с файлами`.
5. Приложение показывает предпросмотр: какие строки будут добавлены, какие попадут в дубли, какие уйдут в конфликты.
6. Сотрудник проверяет отчет по строкам и подтверждает импорт.
7. По мере появления данных сотрудник обновляет почту и оплату абитуриентов отдельным файлом или вручную.
8. Администратор создает или загружает академические группы.
9. Перед переносом открывается `Проверка данных` или `Мастер миграции`.
10. Готовые абитуриенты переносятся в студенты. Готовой считается запись с почтой и отмеченной оплатой.
11. Студенты выгружаются в Excel или используются как итоговый список Moodle.
12. По завершении приемной кампании администратор может архивировать год, чтобы случайно не изменить старые данные.

## Роли и права доступа

В приложении есть несколько ролей. Точный набор доступных кнопок зависит от роли и страницы.

| Роль | Для кого | Что может делать |
| --- | --- | --- |
| `admin` | Администратор системы | Управляет пользователями, правилами логинов, кампаниями, группами, бэкапами, дублями, конфликтами, студентами и всеми записями. |
| `manager` | Куратор процесса | Получает расширенный доступ там, где разрешены действия ассистента. |
| `operator` | Оператор данных | Может работать с массовыми действиями и обновлением данных абитуриентов, где это разрешено. |
| `assistant` | Ассистент приемной комиссии | Может участвовать в миграции и рабочих операциях, где это разрешено. |
| `viewer` | Просмотр | Может смотреть данные без административных действий. |

Регистрация через страницу входа создает заявку на пользователя с ролью `assistant`. Пока администратор не одобрит заявку, войти в рабочую часть нельзя.

Пароли должны быть не короче 8 символов. При сохранении пароль хешируется.

## Что нельзя добавлять в GitHub

В репозиторий не должны попадать локальные данные и секреты:

- `.env` и любые `.env.*`, кроме `.env.example`;
- `.venv`, `venv`, `env` и другие виртуальные окружения;
- `uploads/baze.db` и любые локальные SQLite-базы;
- резервные копии базы;
- загруженные Excel/CSV-файлы;
- сгенерированные Excel-выгрузки;
- логи, временные файлы и кэши;
- `__pycache__`;
- локальные копии проекта, архивы и сборки.

Это уже настроено в `.gitignore`. В папке `uploads` в Git остается только `.gitkeep`, чтобы папка существовала после клонирования.

## Требования

Рекомендуемая среда:

- Windows 10/11 или Linux;
- Python `3.11` 64-bit;
- современный браузер: Chrome, Edge, Firefox или аналогичный;
- доступ в интернет при первой установке зависимостей;
- для Linux-продакшена: `systemd`, Nginx, права `sudo`, желательно отдельный пользователь сервиса.

Зависимости для локального запуска из `requirements.txt`:

```text
Flask>=3.1.3,<4.0
pandas>=2.0
openpyxl>=3.0
xlrd>=2.0
python-dotenv>=1.0
```

Дополнительно для продакшена из `requirements-prod.txt`:

```text
-r requirements.txt
gunicorn>=23.0
```

Почему нужен Python 3.11:

- текущие версии `pandas` и Excel-библиотек нормально поддерживаются на Python 3.11;
- Windows-скрипт `start_one_click.bat` и `install_and_run_windows.ps1` рассчитаны именно на Python 3.11 x64;
- старые установки Python 3.7 часто ломают установку зависимостей.

## Быстрый запуск в Windows

Самый простой вариант для рабочего компьютера Windows:

1. Скачайте или скопируйте папку проекта.
2. Откройте папку проекта.
3. Дважды нажмите:

```text
start_one_click.bat
```

Скрипт сделает следующее:

1. Найдет Python 3.11 64-bit.
2. Если Python 3.11 не найден, попробует установить его через `winget`.
3. Создаст виртуальное окружение `.venv`.
4. Обновит `pip`.
5. Установит зависимости из `requirements.txt`.
6. Создаст папку `uploads`, если ее нет.
7. Создаст `.env`, если его нет.
8. Сгенерирует `SECRET_KEY` и пароль локального администратора.
9. Покажет пароль администратора в окне запуска.
10. Запустит приложение.
11. Покажет адреса для этого компьютера и локальной сети.

Обычно открываются такие адреса:

```text
http://127.0.0.1:5000
http://IP-КОМПЬЮТЕРА:5000
```

Первый адрес нужен на том же компьютере. Второй адрес нужен другим сотрудникам в той же локальной сети.

Если Windows Firewall спросит разрешение, разрешите доступ для частных сетей.

### Если winget отсутствует

Если автоматическая установка Python невозможна:

1. Установите Python 3.11 64-bit вручную: <https://www.python.org/downloads/windows/>.
2. При установке включите `Add Python to PATH`.
3. Снова запустите `start_one_click.bat`.

## Ручная установка в Windows

Откройте PowerShell в папке проекта.

Проверьте Python:

```powershell
python --version
```

Создайте виртуальное окружение:

```powershell
python -m venv .venv
```

Активируйте окружение:

```powershell
.\.venv\Scripts\Activate.ps1
```

Если PowerShell запрещает запуск скриптов:

```powershell
Set-ExecutionPolicy -Scope CurrentUser RemoteSigned
```

Установите зависимости:

```powershell
python -m pip install --upgrade pip
python -m pip install -r requirements.txt
```

Создайте локальные настройки:

```powershell
copy .env.example .env
```

Откройте `.env` и задайте как минимум:

```text
SECRET_KEY=длинная-случайная-строка
ADMIN_DEFAULT_PASSWORD=надежный-пароль-администратора
```

Запустите приложение:

```powershell
python app.py
```

Откройте:

```text
http://127.0.0.1:5000
```

## Ручная установка в Linux

Минимальный локальный запуск:

```bash
python3.11 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
python -m pip install -r requirements.txt
cp .env.example .env
```

Перед первым запуском обязательно измените в `.env` значения `SECRET_KEY` и `ADMIN_DEFAULT_PASSWORD`.

Для локального доступа:

```text
http://127.0.0.1:5000
```

## Первый вход и первичная настройка

При первом запуске приложение создает администратора:

```text
login: admin
password: значение ADMIN_DEFAULT_PASSWORD из .env
```

Если `.env` создан Windows-скриптом, пароль будет сгенерирован автоматически и показан в окне запуска.

После входа администратор должен настроить правила генерации логинов. Пока настройка не завершена, приложение перенаправляет администратора на страницу `Первичная настройка правил логинов`.

На странице настройки можно:

- выбрать стандартные правила;
- задать собственный шаблон логина;
- изменить длину порядкового номера;
- изменить префиксы ошибок и дублей;
- выбрать уникальность логинов внутри кампании или по всей базе;
- настроить поиск года в договоре;
- задать соответствия специальностей и базы.

После сохранения правила применяются к новым импортам и ручному созданию логинов.

## Настройки env

Локальные настройки хранятся в `.env`. Пример находится в `.env.example`.

| Переменная | Пример | Назначение |
| --- | --- | --- |
| `SECRET_KEY` | `replace-with-strong-secret-key-min-32-chars` | Секрет Flask-сессий и CSRF. В продакшене нужен длинный случайный секрет. |
| `ADMIN_DEFAULT_PASSWORD` | `replace-with-strong-admin-password` | Пароль администратора при первом создании пользователя `admin`. |
| `UPLOAD_FOLDER` | `uploads` | Папка для базы, временных файлов, бэкапов и загрузок. |
| `DB_FILENAME` | `baze.db` | Имя SQLite-базы внутри `UPLOAD_FOLDER`. |
| `DEFAULT_CAMPAIGN_YEAR` | `2026` | Кампания по умолчанию. |
| `LEGACY_CAMPAIGN_YEAR` | `2025` | Год для старых записей, где раньше не было поля кампании. |
| `APP_VERSION` | `номер-релиза` | Версия, которая показывается в интерфейсе. Можно обновлять вручную при релизах. |
| `FLASK_ENV` | `production` | Среда Flask. |
| `APP_HOST` | `127.0.0.1` | Адрес, на котором слушает приложение при запуске `python app.py`. |
| `APP_PORT` | `5000` | Порт приложения при запуске `python app.py`. |
| `APP_DEBUG` | `false` | Включение debug-режима. Для рабочих данных должно быть `false`. |
| `MAX_UPLOAD_SIZE_MB` | `16` | Максимальный размер загружаемого файла в мегабайтах. |
| `MAX_CONTENT_LENGTH` | `16777216` | Альтернативное ограничение загрузки в байтах. Если задано, используется вместо расчета от `MAX_UPLOAD_SIZE_MB`. |
| `SESSION_COOKIE_SECURE` | `false` | `true` для HTTPS, чтобы cookie отправлялись только по защищенному соединению. |
| `SESSION_COOKIE_SAMESITE` | `Lax` | SameSite-политика cookie. |
| `ENABLE_HSTS` | `false` | Включение HSTS-заголовка. Используйте только при стабильном HTTPS. |
| `HSTS_MAX_AGE` | `31536000` | Время действия HSTS в секундах. |
| `HSTS_INCLUDE_SUBDOMAINS` | `false` | Добавляет `includeSubDomains` к HSTS. |
| `HSTS_PRELOAD` | `false` | Добавляет `preload` к HSTS. |
| `LOGIN_MAX_ATTEMPTS` | `5` | Сколько неудачных попыток входа разрешено за окно времени. |
| `LOGIN_WINDOW_SECONDS` | `600` | Окно времени для ограничения попыток входа. |

Важно: `.env` нельзя публиковать в GitHub.

## Запуск в локальной сети

Чтобы приложение было доступно с других компьютеров в локальной сети, сервер должен слушать не только `127.0.0.1`.

В `.env` можно указать:

```text
APP_HOST=0.0.0.0
APP_PORT=5000
```

После запуска на других компьютерах откройте:

```text
http://IP-КОМПЬЮТЕРА-С-СЕРВЕРОМ:5000
```

Если страница не открывается:

- проверьте, что компьютер с приложением включен;
- проверьте IP-адрес;
- разрешите приложение в Windows Firewall для частной сети;
- убедитесь, что другой сервис не занял порт `5000`;
- проверьте, что все компьютеры находятся в одной сети или VPN.

## Приемные кампании

Кампания - это год, к которому относятся абитуриенты, дубли и конфликты. Активная кампания выбирается через переключатель на страницах приложения.

Приложение:

- хранит `campaign_year` у абитуриентов, дублей и конфликтов;
- хранит `source_campaign_year` у студентов, которые пришли из абитуриентов;
- пытается определить год по договору, если год явно не указан;
- показывает годы из базового диапазона `2020`-`2030` и годы, найденные в данных;
- позволяет администратору архивировать кампанию.

Архивная кампания:

- доступна для просмотра;
- остается в поиске и отчетах;
- закрыта для изменений, импорта, удаления, ручного редактирования и миграции.

## Форматы файлов

### Файл абитуриентов

Поддерживаются форматы:

- `.xlsx`;
- `.xls`;
- `.csv`.

Обязательные столбцы:

| Столбец | Описание |
| --- | --- |
| `ФИО` | Полное ФИО абитуриента. Первая часть считается фамилией, остальное сохраняется как имя и отчество. |
| `Договор` | Номер договора, по которому система определяет год, специальность и базу. |

Пример:

```csv
ФИО,Договор
Иванов Иван Иванович,2026-ФМ-0001-11
Петров Петр Петрович,2026-СД-0002-9И
```

Шаблон можно скачать в приложении на странице `Работа с файлами`.

### Файл обновления почты и оплаты

Поддерживаются `.xlsx`, `.xls`, `.csv`.

Обязательный столбец:

| Столбец | Описание |
| --- | --- |
| `Договор` | По нему приложение ищет абитуриента в активной кампании. |

Дополнительные столбцы:

| Столбец | Также можно назвать | Описание |
| --- | --- | --- |
| `Email` | `email`, `Почта`, `почта` | Новая почта абитуриента. |
| `Оплата` | `paid`, `Оплачен`, `оплачен` | Статус оплаты. |

Значения оплаты:

```text
да, нет, 1, 0, true, false, yes, no, оплачен, оплачено, не оплачен, не оплачено, paid, unpaid
```

Если почта некорректна, договор не найден или оплата не распознана, приложение покажет отчет по строкам.

### Файл студентов

Поддерживаются `.xlsx`, `.xls`, `.csv`.

Обязательные столбцы:

| Столбец | Описание |
| --- | --- |
| `username` | Логин Moodle. |
| `password` | Пароль Moodle. |
| `email` | Почта студента. |
| `firstname` | Имя. |
| `lastname` | Фамилия. |
| `cohort1` | Академическая группа. |

Необязательные столбцы:

| Столбец | Описание |
| --- | --- |
| `source_dogovor` | Договор при поступлении. Используется в проверках дублей и качества данных. |
| `source_fio` | ФИО из исходной записи при поступлении. |

Пример:

```csv
username,password,email,firstname,lastname,cohort1,source_dogovor,source_fio
26611001,cron,ivanov@example.ru,Иван,Иванов,26ФМ-11-1,2026-ФМ-0001-11,Иванов Иван Иванович
```

Если студент загружен пользователем без административных прав, пароли в карточке и Excel-выгрузке скрываются как `******`.

### CSV групп

Группы можно загрузить CSV-файлом. Поддерживаются кодировки UTF-8 с BOM, Windows-1251 и UTF-8. Разделитель определяется автоматически: `;` или `,`.

Самый простой формат:

```csv
group_name
26ФМ-11-1
26СД-9-1
```

Можно также указать папку года:

```csv
group_year;group_name
2026;26ФМ-11-1
2026;26СД-9-1
```

Альтернативный формат: собрать группу из частей.

```csv
year_code;specialty;base;subgroup
26;ФМ;11;1
26;СД;9;1
```

Приложение понимает такие варианты заголовков:

- для имени группы: `group_name`, `name`, `group`, `группа`, `название`, `название_группы`;
- для года папки: `group_year`, `folder_year`, `year_folder`, `папка`, `год_папки`, `год_групп`, `год_группы`;
- для кода года: `year_code`, `year`, `год`, `год_поступления`;
- для специальности: `specialty`, `spec`, `специальность`, `направление`;
- для базы: `base`, `база`, `база_классов`;
- для подгруппы: `subgroup`, `subgroup_number`, `подгруппа`, `номер_подгруппы`.

## Генерация логинов

Стандартные правила создают логин по шаблону:

```text
{yy}{spec}{base}{seq}
```

Пример:

```text
Договор: 2026-ФМ-0001-11
Логин:   26611001
```

Как читается пример:

- `26` - последние две цифры года;
- `6` - код специальности `ФМ`;
- `11` - код базы `11`;
- `001` - порядковый номер внутри найденной комбинации.

Стандартные коды специальностей:

| Специальность | Код |
| --- | --- |
| `ЛД` | `1` |
| `АД` | `2` |
| `СД` | `3` |
| `СтО` | `4` |
| `СтПр` | `5` |
| `СтП` | `5` |
| `ФМ` | `6` |
| `ЛабД` | `7` |
| `СтД` | `8` |

Стандартные коды базы:

| База в договоре | Код |
| --- | --- |
| `2НМ` | `inm` |
| `2М` | `im` |
| `НМ` | `nm` |
| `М` | `im` |
| `11и` или `11И` | `11i` |
| `9и` или `9И` | `9i` |
| `11` | `11` |
| `9` | `9` |

В собственных правилах доступны поля:

| Поле | Что подставляется |
| --- | --- |
| `{yyyy}` | Полный год, например `2026`. |
| `{yy}` | Последние две цифры года, например `26`. |
| `{spec}` | Код специальности. |
| `{base}` | Код базы. |
| `{seq}` | Порядковый номер с ведущими нулями. |
| `{num}` | Порядковый номер без ведущих нулей. |

Правила также задают:

- `number_width` - длина номера для `{seq}`, от 1 до 8;
- `error_prefix` - префикс временных логинов для конфликтов;
- `duplicate_prefix` - префикс временных логинов для дублей;
- `unique_scope` - уникальность внутри кампании или по всей базе;
- `year_regex` - как искать год в договоре;
- `base_match_mode` - искать базу в последней части договора или в любом месте.

Приложение нормализует договор перед разбором:

- убирает лишние пробелы;
- приводит текст к верхнему регистру;
- заменяет разные виды тире на обычный дефис;
- учитывает похожие латинские и кириллические буквы в договоре.

Если договор нельзя разобрать, запись попадает в конфликты с логином вида `error001`.

## Импорт абитуриентов

Импорт находится на странице `Работа с файлами`.

Алгоритм:

1. Пользователь выбирает файл.
2. Приложение сохраняет его как временный файл в `uploads`.
3. Проверяет обязательные столбцы `ФИО` и `Договор`.
4. Разбирает ФИО на фамилию и имя-отчество.
5. Разбирает договор по текущим правилам логинов.
6. Проверяет договор среди абитуриентов, студентов, дублей и конфликтов текущей кампании.
7. Проверяет повтор договора внутри самого файла.
8. Проверяет возможных тезок по ФИО.
9. Показывает предпросмотр всех строк.
10. Записывает данные в базу только после нажатия `Подтвердить импорт`.
11. Перед записью создает резервную копию базы.
12. После записи отдает Excel-файл `abiturients_with_logins.xlsx`.

Возможные действия по строке:

| Действие | Что значит |
| --- | --- |
| `Будет добавлен` | Строка будет записана в список абитуриентов. |
| `В дубли` | Договор уже ожидает проверки в дублях, строка попадет в `Дублирующие записи абитуриентов`. |
| `Конфликт` | Есть ошибка или риск, строка попадет в `Конфликты логинов`. |

Типовые статусы:

- `Будет добавлен`;
- `Возможный тёзка, договор другой; будет добавлен`;
- `Пустое ФИО`;
- `Ошибка договора`;
- `Договор уже есть у абитуриента`;
- `Договор уже есть у студента`;
- `Договор уже ожидает проверки в дублях`;
- `Договор уже есть в конфликтах`;
- `Договор повторяется в файле импорта`.

## Обновление почты и оплаты

Обновление находится на странице `Работа с файлами`, блок `Обновление почты и оплаты`.

Сценарий:

1. Скачайте шаблон обновлений.
2. Заполните `Договор`.
3. Добавьте `Email`, если нужно обновить почту.
4. Добавьте `Оплата`, если нужно обновить статус оплаты.
5. Загрузите файл.
6. Приложение обновит только найденные записи активной кампании.
7. Ошибки и ненайденные договоры будут показаны в отчете.

Перед применением обновлений создается резервная копия базы.

## Загрузка студентов

Загрузка студентов находится на странице `Работа с файлами`, блок `Загрузка студентов`.

Алгоритм:

1. Пользователь загружает файл студентов.
2. Приложение проверяет обязательные поля.
3. Проверяет email.
4. Проверяет, существует ли такой `username` в базе студентов.
5. Проверяет повторы `username` внутри файла.
6. Показывает предпросмотр.
7. Записывает новых студентов только после подтверждения.
8. Повторные логины отправляет в `students_duplicates`.
9. Строки с незаполненными обязательными полями или неправильной почтой пропускает.
10. Перед записью создает резервную копию базы.

Итоговые категории:

| Категория | Что происходит |
| --- | --- |
| `Будет добавлен` | Студент будет создан в таблице `students`. |
| `В дубли` | Строка будет сохранена в `students_duplicates`. |
| `Пропущен` | Строка не будет записана из-за ошибки. |

## Академические группы

Академические группы управляются в админке на странице `Академические группы`.

Формат группы:

```text
26ФМ-11-1
```

Где:

- `26` - год группы;
- `ФМ` - специальность;
- `11` - база;
- `1` - номер подгруппы.

Приложение:

- хранит группы по папкам года, например `2026`;
- показывает заполненность каждой группы;
- считает группу заполненной при 25 студентах;
- предлагает создать следующую подгруппу, когда текущая заполнена;
- умеет принудительно создать следующую подгруппу;
- позволяет скрыть группу, чтобы ее нельзя было выбрать при миграции;
- не удаляет группу, если в ней есть студенты;
- загружает группы из CSV;
- дает скачать CSV-шаблон.

Год группы должен соответствовать папке года. Например, в папке `2026` группа должна начинаться с `26`.

## Миграция абитуриентов в студенты

Миграция переносит абитуриентов в таблицу студентов Moodle.

Абитуриент переносится только если:

- у него заполнена почта;
- у него отмечена оплата;
- его логин еще не существует в таблице студентов;
- выбранная группа существует и не скрыта;
- активная кампания не архивирована.

При переносе создается студент:

| Поле студента | Откуда берется |
| --- | --- |
| `username` | Логин абитуриента. |
| `password` | Значение `cron`. |
| `email` | Почта абитуриента. |
| `firstname` | Имя и отчество из абитуриента. |
| `lastname` | Фамилия абитуриента. |
| `cohort1` | Выбранная академическая группа. |
| `source_campaign_year` | Кампания абитуриента. |
| `source_dogovor` | Договор абитуриента. |
| `source_fio` | Исходное ФИО абитуриента. |

После успешного переноса запись удаляется из списка абитуриентов и появляется в списке студентов.

Если в выбранной группе не хватает мест:

- без авторазбиения приложение попросит выбрать или создать следующую подгруппу;
- с авторазбиением приложение само распределит людей по текущей и следующим подгруппам.

Перед переносом создается резервная копия базы.

## Дубли и конфликты

### Дубли абитуриентов

Дубли хранятся отдельно в `pending_duplicates`.

Администратор может:

- подтвердить дубль, тогда запись станет абитуриентом;
- отклонить дубль, тогда запись удалится из очереди;
- отклонить все дубли текущей кампании.

### Конфликты логинов

Конфликты хранятся в `login_conflicts`.

Запись попадает в конфликты, если:

- ФИО пустое;
- договор не удалось разобрать;
- договор уже есть у абитуриента;
- договор уже есть у студента;
- договор уже есть в конфликтах;
- договор повторяется внутри файла импорта;
- система не может безопасно назначить логин.

Администратор может открыть конфликт, вручную указать корректный логин и перенести запись в абитуриенты. Перед разрешением конфликта создается резервная копия.

## Проверка качества данных

Страница `Проверка данных` группирует проблемы по текущей кампании.

Проверки абитуриентов:

- нет почты;
- не отмечена оплата;
- некорректная почта;
- нет договора;
- повторяющиеся договоры;
- одинаковое ФИО.

Проверки студентов:

- нет академической группы;
- нет договора при поступлении;
- нет кампании поступления;
- некорректная почта;
- повторяющиеся договоры.

Проверки миграции:

- есть дублирующие записи абитуриентов;
- есть конфликты логинов.

Панель состояния использует эти проверки, чтобы показать центр задач: что именно нужно исправить перед миграцией.

## База данных и миграции схемы

По умолчанию база находится здесь:

```text
uploads/baze.db
```

SQLite-таблицы:

| Таблица | Назначение |
| --- | --- |
| `abiturients` | Абитуриенты текущих и прошлых кампаний. |
| `pending_duplicates` | Дубли абитуриентов, ожидающие решения. |
| `login_conflicts` | Конфликты логинов и ошибочные строки импорта. |
| `students` | Студенты Moodle. |
| `students_duplicates` | Дубли при загрузке студентов. |
| `groups` | Академические группы по годам. |
| `users` | Пользователи приложения. |
| `login_attempts` | Неудачные попытки входа для ограничения перебора паролей. |
| `audit_logs` | Журнал действий. |
| `campaign_settings` | Настройки кампаний, включая архив. |
| `login_generation_settings` | Правила генерации логинов. |

При запуске приложение само вызывает инициализацию базы:

- создает недостающие таблицы;
- добавляет новые столбцы в старую базу;
- переносит старую таблицу `students`, если раньше она использовалась как список абитуриентов;
- добавляет `campaign_year` в старые записи;
- добавляет поля `paid`, `source_campaign_year`, `source_dogovor`, `source_fio`;
- добавляет год и статус скрытия групп;
- создает индексы;
- переводит старые незахешированные пароли пользователей в хеши.

Перед обновлением рабочей базы все равно сделайте резервную копию.

## Резервные копии и журнал действий

Бэкапы создаются в папке `uploads` с именем вида:

```text
baze_backup_<reason>_<timestamp>.db
```

Приложение автоматически создает резервную копию перед такими операциями:

- импорт абитуриентов;
- обновление почты и оплаты;
- загрузка студентов;
- миграция абитуриентов в студенты;
- массовые изменения абитуриентов;
- удаление абитуриента;
- редактирование абитуриента;
- разрешение конфликта логина;
- удаление студента;
- изменение статуса архива кампании;
- удаление пользователя;
- очистка абитуриентов текущей кампании;
- восстановление базы из бэкапа.

На странице `Резервные копии` администратор может:

- посмотреть список бэкапов;
- скачать файл бэкапа;
- восстановить базу из выбранной копии.

Перед восстановлением создается дополнительный бэкап текущего состояния.

Журнал действий хранит:

- пользователя;
- действие;
- тип сущности;
- идентификатор сущности;
- детали;
- IP-адрес;
- время операции.

## Безопасность

В проекте реализованы базовые меры защиты:

- пароли пользователей хранятся через `werkzeug.security.generate_password_hash`;
- старые незахешированные пароли мигрируют при запуске;
- вход защищен от перебора через `LOGIN_MAX_ATTEMPTS` и `LOGIN_WINDOW_SECONDS`;
- формы защищены CSRF-токенами;
- загружаемые файлы проверяются по расширению;
- временные файлы создаются только внутри `UPLOAD_FOLDER`;
- скачивание временных файлов и бэкапов использует проверку безопасного пути;
- включены `SESSION_COOKIE_HTTPONLY` и `SESSION_COOKIE_SAMESITE`;
- можно включить `SESSION_COOKIE_SECURE` для HTTPS;
- добавляются заголовки `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`;
- можно включить HSTS через `.env`;
- `.env`, база, бэкапы и загрузки исключены из Git.

Для рабочей установки:

- смените пароль администратора после первого входа;
- используйте длинный `SECRET_KEY`;
- не включайте `APP_DEBUG=true` на реальных данных;
- храните `.env` и `uploads/baze.db` вне публичного доступа;
- делайте регулярные бэкапы;
- используйте HTTPS, если доступ идет не только внутри доверенной локальной сети;
- ограничьте доступ к серверу на уровне сети или VPN.

## Linux-развертывание через systemd и Nginx

В проекте есть готовые файлы:

```text
deploy.sh
update.sh
deploy/manticore.service
deploy/nginx.conf
deploy/nginx-https.example.conf
```

Базовое развертывание:

```bash
sudo bash deploy.sh /srv/manticore example.com
```

Что делает `deploy.sh`:

1. Копирует проект в `/srv/manticore` или указанную папку.
2. Создает `.env`, если его нет.
3. Генерирует `SECRET_KEY` и пароль администратора.
4. Создает виртуальное окружение.
5. Устанавливает `requirements-prod.txt`.
6. Создает папку `uploads`.
7. Настраивает владельца и права.
8. Создает systemd-сервис `manticore`.
9. Запускает приложение через `gunicorn` на `127.0.0.1:8000`.
10. Настраивает Nginx как reverse proxy.
11. Перезапускает сервисы.

После успешного деплоя приложение доступно по:

```text
http://example.com
http://IP-СЕРВЕРА
```

Для HTTPS используйте `deploy/nginx-https.example.conf` как пример и настройте сертификаты, например через Let's Encrypt.

Полезные команды:

```bash
sudo systemctl status manticore
sudo systemctl restart manticore
sudo journalctl -u manticore -f
sudo nginx -t
sudo systemctl reload nginx
```

## Обновление приложения

Обновление не должно перезаписывать локальные данные: `.env`, папка `uploads` и база `uploads/baze.db` исключены из Git. Перед изменением кода скрипт создает резервную копию базы в папке данных.

Для Windows:

Перед обновлением закройте окно, в котором сейчас запущено приложение.

```powershell
.\update_windows.bat
```

Если нужно сразу запустить приложение после обновления:

```powershell
.\update_and_run_windows.bat
```

Для Linux-сервера с systemd:

```bash
sudo bash update.sh /srv/manticore
```

Для локального Linux или macOS из папки проекта:

```bash
bash update.sh
```

Скрипты:

1. Переходит в папку приложения.
2. Создает SQLite-бэкап базы, если база уже существует.
3. Если там есть Git-репозиторий, выполняет `git pull --ff-only`.
4. Обновляет зависимости в виртуальном окружении.
5. На Linux-сервере с systemd перезапускает сервис `manticore`.
6. Если установлен Nginx, проверяет конфигурацию и перезагружает Nginx.

Если в рабочей копии есть локальные изменения, Git не будет перезаписывать их молча: при конфликте обновление остановится после создания бэкапа базы.

## Типовые проблемы

### Не создается администратор

Проверьте, что в `.env` задано:

```text
ADMIN_DEFAULT_PASSWORD=...
```

Если пользователя `admin` еще нет, приложение не запустится без этого значения.

### Не получается войти

Проверьте:

- правильность логина и пароля;
- что заявка пользователя одобрена администратором;
- что не превышен лимит попыток входа;
- что используется актуальная база `uploads/baze.db`.

### После нескольких ошибок вход временно заблокирован

По умолчанию разрешено 5 неудачных попыток за 600 секунд. Настройки:

```text
LOGIN_MAX_ATTEMPTS=5
LOGIN_WINDOW_SECONDS=600
```

### Файл не загружается

Проверьте:

- расширение файла: `.xlsx`, `.xls`, `.csv`;
- размер файла и `MAX_UPLOAD_SIZE_MB`;
- наличие обязательных столбцов;
- что файл не открыт в Excel в режиме блокировки;
- что активная кампания не архивирована.

### Импорт абитуриентов отправляет строки в конфликты

Чаще всего причина в договоре:

- в договоре нет года;
- специальность не входит в настройки кодов;
- база не найдена в последней части договора;
- договор уже есть в абитуриентах, студентах или конфликтах;
- договор повторяется внутри файла.

Откройте `Правила логинов` и проверьте коды специальностей, коды базы и режим поиска базы.

### Абитуриент не переносится в студенты

Проверьте:

- заполнена ли почта;
- отмечена ли оплата;
- нет ли такого логина в списке студентов;
- выбрана ли видимая академическая группа;
- есть ли свободное место в группе;
- не архивирована ли кампания.

### Группа не создается

Проверьте формат:

```text
26ФМ-11-1
```

И соответствие папке года. В папке `2026` группа должна начинаться с `26`.

### Нужна следующая подгруппа

Если группа заполнена, приложение покажет следующую подгруппу. Ее можно создать на странице `Академические группы` или включить авторазбиение при миграции.

### Нужно откатиться назад

Откройте `Панель администратора -> Резервные копии`, выберите нужный бэкап и восстановите базу. Перед восстановлением приложение создаст дополнительный бэкап текущего состояния.

### README показывает старую версию приложения

Версия в интерфейсе берется из `APP_VERSION` в `.env`. Если вы обновили код, но хотите видеть новый номер версии в подвале приложения, обновите `APP_VERSION` вручную.
